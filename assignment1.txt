An introduction to non-blocking communicationNon-blocking TCP communication means the
socket doesn’t wait for operations to finish. Instead, recv() or send() return immediately, even if the buffer
is full or empty, no data is available or only partial data is obtained. An error exception may occured
likeBlockingIOError. The blocking sockets pause your program until the operation finishes, while
non-blocking sockets let your program continue and handle readiness explicitly.
An exampleImagine a cashier who serves one customer at a time and refuses to look up until that
customer is finished. If 100 customers arrive, you would need 100 cashiers.
Programming in blocking mode results in simpler code, but the program must wait as I/O operations are
performed sequentially. Non-blocking mode runs in high performance servers, asynchronous I/O. It allows
a single thread to handle thousands of concurrent connections efficiently. This design improves scalability,
performance, and responsiveness compared to traditional blocking I/O models

1.3 Server Processes
ProxyThe proxy acts as an intermediary between the clients and the backend or webapp services. In
this architecture, it receives incoming requests from multiple clients and forwards them to the appropriate
backend resources.
BackendThe backend refers to the core services or servers that handle the actual processing of client
requests. The ”previous step” proxy process routes the client requests to these backend units based on
availability, load, or specific routing rules, ensuring efficient and reliable service delivery.
WebApp - AsynapRousThe WebApp, labeled “AsynapRous” (short for Asynchronous App Routing)
in the image, represents a specialized group of web-facing applications or services. These are typically
user-facing components such as web servers or frontend applications that deliver content and interfaces to
clients. The AsynapRous may be optimized for rendering HTML, serving static assets, or interacting with
users directly through RESTful. Like the backend, it receives traffic through the proxy, which helps manage
access and performance. AsynapRous can be run in independent mode without any proxy

2.1 Implement non-blocking mechanims
The implementation of non-blocking mechanisms relies on operating system services that allow I/O oper-
ations to be halted immediately rather than blocking execution. However, the responsibility for correctly
managing ongoing communication and ensuring consistency is delegated to the application level. As a result,
developers must design and implement appropriate handling logic to guarantee correctness and reliability in
non-blocking communication system.
When a communication is halted, there are multiple processing flows to be handled.
•Multi-thread create a thread to handle the new created processing flows.
•Event-driven relies on registering callbacks that are executed when an operation completes. Instead
of waiting for I/O, the program continues running, and the event loop triggers the callback once the
communication is completed nad the data is ready. While powerful, it can lead to complex nested code
structures, often referred to as “callback hell.”
–Some recently updated event mechansims (i.e futures or promises) allow chaining events by putting
a placeholder for object representing a result that will be available later.
–Event loop pub/sub puts a simple loop and use socket in a loop listens for events and dispatches
them to handlers. Dive in the ’real’ code and you will see the simple socket send()/recive() with
some noisy wrappers.
•Coroutine (or asyncio, async/await) represents a more readable approach to asynchronous I/O. The
asyncio event loopschedules and runs these coroutines, pausing them when they wait for I/O and
resuming them once the operation completes.
– asynciois the framework for writing concurrent code using async/await syntax.
∗*StreamReaderandStreamWriterare specific high-level APIs withinasyncioused for
handling I/O operations, i.e. network communication. They are components of theasyncio.
–A loopasyncio.run()to start the event loop and run a coroutine until completion
–The management mechanism asynchronous programs to runs your async tasks to completion and
then clean up as in the AsyncIO runner1.

Node.js vs BunNode.js started as a single-threaded runtime that relied heavily on callbacks, and later
added coroutine-like features through async/await. Bun, by contrast, is designed with multi-threading at its
core, which can deliver higher performance but also requires more management effort and system resources.
Node.js vs Nest.jsNest is a Backend Server Framework which can be used to build up powerful APIs.

2.2 Implement the authentication for HTTP server
DescriptionTo authenticate a user, there are two common approaches include HTTP headers and Cookies.
The former RFC 2617 or the updated version rfc 7235 with extended proxy usecase, specifies HTTP use
the header “WWW-Authenticate” to request for an authentication; then, the browser responds with an
“Authorization” header that carries user login details. The latter RFC 6265, uses cookies to store with
“Set-Cookie” header when a user log in successfully. The server sends back a instruction that stores a small
piece of data in user browser. From then on, The browser automatically includes that cookie in every request,
so the server knows it’s still that logged-in user.

class AsynapRous (builtins.object )
   The fully mutable :class:` AsynapRous  <AsynapRous >` object, which is a lightweight,
mutable web application router for deploying RESTful URL endpoints.
 
The `AsynapRous ` class provides a decorator-based routing system for building simple
RESTful web applications.  The class allows developers to register route handlers 
using decorators and launch a TCP-based backend server to serve RESTful requests. 
Each route is mapped to a handler function based on HTTP method and path. It mappings
supports tracking the combined HTTP methods and path route mappings internally.
 
Usage::

  >>> app = AsynapRous ()

  >>> def login(headers="guest", body="anonymous"):
  >>>     return {'message': 'Logged in'}
 
  >>> @app. route('/hello', methods=['GET'])
  >>> def hello(headers, body):
  >>>     return {'message': 'Hello, world!'}
 
  >>> app. run()
